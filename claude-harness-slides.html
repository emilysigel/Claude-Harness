The Problem

Out of the Box vs. Configured

Claude Code without configuration is like a highly capable new hire with no onboarding — talented, but no guardrails, no process, no shared expectations.

Unconfigured Claude Code

  • No shared plan — can drift mid-task
  • Can run destructive commands
  • No structured review built in
  • Behavior varies session to session
  • Hard to enforce team standards

Configured Claude Code

  • Shared plan as source of truth
  • Guardrails block risky operations
  • Consistent review process every time
  • Predictable, repeatable workflow
  • Team standards enforced at runtime
Insurance context: When you're touching policy calculations, claims logic, or compliance-sensitive code — you want guardrails, not just good intentions.
Today's Example

Meet Claude Harness

Claude Harness is a plugin for Claude Code that packages up a structured development workflow. Think of it as a pre-built configuration you can install, use as-is, or adapt. We're using it today as a concrete example of what configuration can look like.

🔌

It's a plugin

Installed via Claude Code's plugin marketplace. One command to add it to any project.

🏗️

It adds structure

Introduces a planning file, a review step, and a release step as named commands.

🛡️

It enforces safety

A Go-native engine intercepts tool calls and blocks destructive operations before they run.

🔄

It's a pattern, not a product

You could build similar structure yourself — Harness shows what "configured" looks like.

Under the Hood

How Claude Code Configuration Works

Claude Code exposes several extension points. Harness uses all of them — understanding them helps you configure Claude Code for your needs.

📋

Skills (slash commands)

Named commands like /harness-plan. You type them, Claude runs a defined workflow.

e.g. /generate-claims-summary

🪝

Hooks

Shell scripts or binaries that run before/after every tool call. Used for guardrails and logging.

e.g. block writes to prod config

🤖

Agents

Sub-agents Claude can spin up to handle specific roles — reviewer, planner, implementer.

e.g. a compliance review agent

⚙️

Settings & Permissions

What Claude is allowed to do, which commands auto-approve, and what requires confirmation.

e.g. auto-approve read, ask for writes

The Harness Workflow

A Structured Loop: 5 Verbs

Harness collapses a complex workflow into five named commands. Each one hands off to the next — and you stay in control at every step.

0
/harness-setup
Bootstrap project files & config
›
1
/harness-plan
Idea → Plans.md with acceptance criteria
›
2
/harness-work
Parallel implementation with self-check
›
3
/harness-review
4-angle code review (security, perf, quality, a11y)
›
4
/harness-release
CHANGELOG, tag & handoff
Key insight: You approve the plan before any code is written. The loop can't start without your sign-off. This is what "human in the loop" looks like in practice.
Live Demo Setup

Installing Harness — 3 Commands

Start Claude Code inside your project folder, then run these three commands. That's the entire setup.

# Step 1 — add the plugin source /plugin marketplace add Chachamaru127/claude-code-harness # Step 2 — install it /plugin install claude-code-harness@claude-code-harness-marketplace # Step 3 — initialize this project /harness-setup

What setup does: Creates a Plans.md file, wires up the hooks, and registers the 5 verb commands for this project. No Node.js needed — runs on a single Go binary.

Safety Configuration

Guardrails: What Gets Blocked

Harness installs a hook engine that intercepts every tool call before Claude acts. These run in under 10ms — invisible to you, always on.

DENY sudo commands
DENY Writes to .git, .env, secret files
DENY git push --force
DENY git reset --hard main/master
DENY --no-verify / skip hooks
ASK Writes outside the project folder
ASK rm -rf
ASK Direct push to main / master
WARN Edits to protected files
WARN Skipped tests / assertion tampering
AUTO ✓ Read-only ops (git status, tests)
AUTO ✓ Standard build commands
Built-in Quality Gate

/harness-review — Before You Ship

Every review runs four independent perspectives. This is a sub-agent doing the review — not Claude second-guessing itself, but a separate analysis pass.

🔒

Security

Injection vulnerabilities, auth issues, secret exposure.

Critical for policy & PII handling

🚀

Performance

Bottlenecks, memory usage, scaling risks.

Matters for batch claims processing

🎯

Quality

Maintainability, naming, test coverage signals.

Auditors read this code too

♿

Accessibility

WCAG compliance, screen reader support.

Required for customer-facing portals

Adapting This to Your Projects

Harness is an example — not a requirement. Here's how to think about building your own configuration.

Start with Harness as-is

  • Install it, run /harness-setup
  • Try /harness-plan on a real task
  • See what the review step catches
  • Use it for 1 sprint, note what helps

Then customize

  • Add your own slash commands (skills)
  • Write hooks for your protected files
  • Create an insurance-specific review agent
  • Add auto-allow rules for your safe ops

Team adoption

  • Check config into the repo — it travels with code
  • Share your Plans.md template
  • Document your guardrail decisions
  • Review sessions become team knowledge

Update when needed

  • /plugin update claude-code-harness
  • Or fork and maintain your own version
  • Config is code — PR it like anything else
  • Claude Code v2.1+ for all features
