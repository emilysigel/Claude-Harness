Configuration Tool #1

CLAUDE.md — Persistent Instructions

# CLAUDE.md ## Project context This is a policy rating engine. Never modify rate tables directly. All changes need a test in /tests/rates/. ## Coding standards - Python 3.11, type hints required - No hardcoded dollar amounts - Log all decisions to audit_trail.log ## What to avoid - Do not commit to main directly - Do not read files in /data/pii/ - Ask before touching claims_core.py

What it does

  • Injected into every Claude conversation in this project
  • Gives Claude persistent context — no re-explaining every session
  • Enforces conventions, naming, patterns, and no-go zones
  • Checked into your repo — travels with the code
File locations CLAUDE.md — project-level (checked in)
~/.claude/CLAUDE.md — global (applies everywhere)
Insurance example Document which files are PII-sensitive, which functions touch regulated calculations, and what the audit trail requirements are.
Configuration Tool #2

settings.json — Permissions & Behavior

// .claude/settings.json { "permissions": { "allow": [ "Bash(git status)", "Bash(npm test:*)", "Read(**)" ], "deny": [ "Bash(sudo:*)", "Write(/data/pii/**)" ] }, "model": "claude-opus-4-7", "env": { "LOG_LEVEL": "audit" } }

What it does

  • Controls what Claude can do without asking
  • Sets which operations require your confirmation
  • Hard-denies operations that should never run
  • Sets the model, env vars, and other behavior
File locations .claude/settings.json — project (check in)
~/.claude/settings.json — global defaults
Insurance example Auto-allow read + test commands. Deny writes to /data/pii/ and prod config files entirely.
Configuration Tool #3

Hooks — Run Code Around Every Tool Call

// .claude/hooks.json { "hooks": { "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command", "command": "./scripts/guardrail.sh" }] }], "PostToolUse": [{ "matcher": "Write", "hooks": [{ "type": "command", "command": "./scripts/audit-log.sh" }] }] } }

What it does

  • Runs your script before or after any tool call
  • Hook can block the action, modify it, or just log it
  • Matched by tool type (Bash, Write, Read, etc.)
  • The script can be anything: shell, Python, a Go binary
Hook timing options PreToolUse — intercept before Claude acts
PostToolUse — react after the action
SessionStart — run when session opens
PreCompact — pause before context compresses
Insurance example PreToolUse hook checks if a Bash command touches claims_core.py — if yes, logs to audit trail and prompts for approval.
Configuration Tool #4

Agents — Specialized Sub-Agents

// agents/compliance-reviewer.md ## Role You are a compliance review agent for insurance software. Your only job is to review code changes for regulatory risk. ## Review checklist - Does this touch PII fields? - Are audit log calls present? - Does this bypass any rate-approval flow? - Is error handling present for all external data calls? ## Output Return PASS, WARN, or FAIL with specific line references.

What it does

  • A separate Claude instance with its own instructions
  • Spun up by the main agent for specialized work
  • Can have its own permissions and model
  • Independent — won't rationalize its own prior work
Common agent roles Reviewer · Planner · Implementer · Tester · Summarizer
Insurance example A dedicated compliance reviewer agent that only checks for PII handling, audit logging, and regulatory bypasses — called automatically before any commit.
Putting It Together

How the Four Tools Work Together

Each tool controls a different layer of Claude's behavior. Together they give you a fully configured environment.

📄

The what — project context, conventions, and constraints Claude always knows.

Persistent knowledge layer

⚙️

settings.json

The allowed — which operations run freely, which need approval, which are blocked entirely.

Permission control layer

🪝

Hooks (JSON)

The around — scripts that intercept every tool call for guardrails, logging, or enrichment.

Runtime enforcement layer

🤖

Agents

The who else — specialized sub-agents for review, planning, or domain-specific analysis.

Specialization layer

You don't need all four on day one. Start with CLAUDE.md (free, instant value), then add settings → hooks → agents as your needs grow.
Why It Matters

Out of the Box vs. Configured

An unconfigured Claude Code is like a capable new hire with no onboarding — talented, but no guardrails, no process, no shared expectations.

Unconfigured

  • No persistent context — re-explain every session
  • Can run destructive commands
  • No structured review built in
  • Behavior varies session to session
  • Hard to enforce team standards

Configured

  • CLAUDE.md carries context automatically
  • settings.json blocks risky operations
  • Hooks enforce rules on every tool call
  • Agents handle review consistently
  • Team standards are in code, not memory
Insurance context: When you're touching policy calculations, claims logic, or compliance-sensitive code — you want guardrails in code, not just good intentions.
Worked Example

Claude Harness — A Pre-Built Configuration

Claude Harness is a plugin that uses all four configuration tools to package a complete development workflow. We're not adopting it wholesale — we're using it to see what a fully configured setup looks like in practice.

📄

Uses CLAUDE.md

Injects workflow conventions, plan format, and review expectations into every session.

⚙️

Uses settings.json

Auto-allows safe ops (git status, tests). Blocks force-push, sudo, and secret writes.

🪝

Uses hooks

A Go binary runs in under 10ms on every tool call — enforcing 13 guardrail rules.

🤖

Uses agents

Separate reviewer, planner, and implementer agents — each with its own role and instructions.

Example: Hooks in Action

What Guardrails Look Like at Runtime

Harness ships a Go binary as its hook. It runs on every tool call in under 10ms — invisible when safe, blocking when not.

DENY sudo commands
DENY Writes to .git, .env, secret files
DENY git push --force
DENY git reset --hard main/master
DENY --no-verify / skip hooks
ASK Writes outside the project folder
ASK rm -rf
ASK Direct push to main / master
WARN Edits to protected files
WARN Skipped tests / assertion tampering
AUTO ✓ Read-only ops (git status, tests)
AUTO ✓ Standard build commands

You can write the same pattern yourself — a shell script or binary that exits non-zero to block, zero to allow.

Take This Home

Where to Start

You don't need a plugin to get started. Build up from the simplest layer.

Day 1 — CLAUDE.md

  • Create CLAUDE.md in your project root
  • Add project context + what to avoid
  • List your protected files by name
  • Zero setup cost, immediate value

Week 1 — settings.json

  • Create .claude/settings.json
  • Auto-allow your safe read/test commands
  • Deny writes to sensitive paths
  • Check it into your repo

Month 1 — Hooks

  • Write a PreToolUse shell script
  • Start with just logging — no blocking yet
  • Add blocking rules once you trust the pattern
  • Or install a plugin like Harness for a head start

Month 2+ — Agents

  • Write a compliance-reviewer agent prompt
  • Call it from your skills or hooks
  • Build a team-specific review checklist
  • Config is code — PR it like anything else
